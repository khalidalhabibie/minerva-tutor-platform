# API Design

## API Style

- Base path: `/api`.
- Authentication: JWT bearer token in `Authorization: Bearer <token>`.
- Content type: JSON except multipart upload endpoints.
- API documentation: Swagger/OpenAPI generated by NestJS decorators.
- Errors should use consistent JSON responses and avoid stack traces.

## Roles

- `PARENT`: can manage own tuition cases, invite or revoke tutors, upload case documents, browse tutor profiles.
- `TUTOR`: can manage own tutor profile, upload supporting documents, and view only invited tuition cases.

## Core Resources

### User

Fields exposed to clients:

- `id`
- `email`
- `role`
- `createdAt`

Never expose password hashes.

### Tuition Case

Expected fields:

- `id`
- `parentId`
- `title`
- `description`
- `subject`
- `level`
- `location`
- `budget`
- `status`
- `createdAt`
- `updatedAt`

Only the owning parent can mutate a case. Tutors can view a case only through an active invitation.

### Case Invitation

Expected fields:

- `id`
- `caseId`
- `tutorId`
- `status`
- `createdAt`
- `revokedAt`

Use statuses such as `ACTIVE` and `REVOKED`. Revoked invitations must not grant access.

### Document

Expected fields exposed to clients:

- `id`
- `fileName`
- `mimeType`
- `sizeBytes`
- `createdAt`
- `uploadedById`

Do not expose local filesystem paths or internal storage keys unless the key is not sensitive and cannot reveal path structure.

### Tutor Profile

Expected fields:

- `id`
- `userId`
- `displayName`
- `bio`
- `subjects`
- `hourlyRate`
- `experienceYears`
- `createdAt`
- `updatedAt`

The tutor owns their profile. Parents can browse public profile fields.

## Endpoint Outline

### Auth

- `POST /auth/register`
  - Optional for MVP if seeded users are sufficient.
  - Creates a Parent or Tutor user.

- `POST /auth/login`
  - Accepts email and password.
  - Returns access token and user summary.

- `POST /auth/logout`
  - Returns success.
  - For MVP, frontend discards the token.

- `GET /auth/me`
  - Returns current authenticated user.

### Parent Tuition Cases

- `POST /cases`
  - Parent only.
  - Creates a case owned by the current parent.

- `GET /cases`
  - Parent: lists own cases.
  - Tutor: lists invited cases.

- `GET /cases/:caseId`
  - Parent: own case only.
  - Tutor: active invitation only.

- `PATCH /cases/:caseId`
  - Parent owner only.
  - Updates editable case fields.

### Case Invitations

- `POST /cases/:caseId/invitations`
  - Parent owner only.
  - Invites a tutor to the case.

- `GET /cases/:caseId/invitations`
  - Parent owner only.
  - Lists invitations for the case.

- `DELETE /cases/:caseId/invitations/:invitationId`
  - Parent owner only.
  - Revokes tutor access.

### Case Documents

- `POST /cases/:caseId/documents`
  - Parent owner or actively invited tutor.
  - Multipart upload.

- `GET /cases/:caseId/documents`
  - Parent owner or actively invited tutor.
  - Lists document metadata.

- `GET /cases/:caseId/documents/:documentId/download`
  - Parent owner or actively invited tutor.
  - Re-checks access before streaming file.

### Tutor Profiles

- `GET /tutors`
  - Parent only.
  - Lists public tutor directory records.

- `GET /tutors/:tutorId`
  - Parent only, or tutor viewing own profile if useful.
  - Returns public profile details.

- `GET /tutor-profile/me`
  - Tutor only.
  - Returns own profile.

- `PUT /tutor-profile/me`
  - Tutor only.
  - Creates or updates own profile.

- `POST /tutor-profile/me/documents`
  - Tutor only.
  - Uploads supporting documents.

- `GET /tutor-profile/me/documents`
  - Tutor only.
  - Lists own supporting documents.

## Response and Error Conventions

Success responses should return resource objects directly or arrays for list endpoints.

Recommended error shape:

```json
{
  "statusCode": 403,
  "message": "You do not have access to this resource",
  "error": "Forbidden"
}
```

Use:

- `400` for validation errors.
- `401` for missing or invalid authentication.
- `403` for authenticated users without permission.
- `404` when a resource does not exist or should not be revealed.
- `409` for duplicate invitations or conflicting state.

## Swagger Requirements

- Document every endpoint with summary, auth requirement, request DTO, and response DTO.
- Mark protected endpoints with bearer auth.
- Include upload endpoints as multipart form data.
- Include common error responses for `400`, `401`, `403`, and `404`.
